¿Que es KitHack?

Framework diseñado para automatizar la instalación de distintas herramientas enfocadas al pentesting. Mediante una recopilación se obtuvo un listado de las herramientas más frecuentadas por los amantes de la ciberseguridad, y de la misma forma se obtuvieron herramientas que son poco conocidas pero que funcionan correctamente. Todas estas se clasificaron de la siguiente forma:

  • Android
  • Windows
  • Phishing
  • Wifi Attacks
  • Passwords Attacks
  • Web Attacks
  • Spoofing
  • Information Gathering
  • Others

En cada sección se encuentran las diferentes herramientas que podemos instalar junto con una pequeña descripción que nos beneficia en saber de que tratan o que funciones tienen.

¿Cómo es el proceso de instalación de las herramientas?

KitHack de manera automatizada se encarga tanto de la clonación del repositorio como de la instalación de sus dependencias o requerimientos especiales. Una vez terminada la instalación de su herramienta, esta se almacenará en la carpeta tools, y dentro de esta carpeta nos vamos a encontrar con otras subcarpetas con los nombres de cada categoría.
Por ejemplo, si la herramienta que usted eligió se encuentra dentro de la categoría Android, esta se almacenará en el directorio KitHack/tools/Android/mi_herramienta.

Si por alguna razón usted inconscientemente trata de volver a instalar una herramienta que ya instaló anteriormente, KitHack va a detectar que esa herramienta ya existe y le arrojará un mensaje indicandole esto mismo y lo regresará al menú principal.

Generador de backdoors.

Una de las funciones más atractivas que tiene KitHack es el generador de puertas traseras que funciona mediante Metasploit-Framework. Este consiste en la generación de puertas traseras tanto para Linux, Windows, Android y Mac OS. Tambien contiene 4 formatos multiplataforma los cuales son; PHP, python, bash y perl.

Las 3 principales caracteristicas que tiene este generador de puertas traseras son las siguientes:

  • Unificación de los 2 tipos de Payloads.
  • Utilización de la herramienta Ngrok para generar conexiones TCP (puerto 443 por default).
  • Incorporación de un nuevo método para la certificación de APKS maliciosas.

Tipos de payloads.

Metasploit en su conjunto de herramientas nos proporciona una gran variedad de payloads los cuales se clasifican por 2 tipos:

  • Payloads por etapas: Este tipo de payload se mantiene de manera muy compacta y solo se encarga de proporcionar los medios para realizar los escenarios de explotación.
    Cada vez que utilizamos windows/meterpreter/reverse_tcp, linux/meterpreter/reverse_tcp, android/meterpreter/reverse_tcp y entre otros, estamos indicandole a metasploit que prepare un tipo de payload que se divida en 2 etapas. Este en varios escenarios de explotación puede funcionar muy bien. Sin embargo, existen otros escenarios donde este tipo de payload es lo menos ideal a realizar por el tamaño de memoria que utilizan los bufer y los DLL.
    El ejemplo más común de este tipo de escenarios es donde nosotros como atacantes nos encontramos en entornos de bajo ancho de banda o de alta latencia y queremos realizar un ataque fuera de nuestra red local. El costo de datos o tiempo de carga metsrv, stdapi y priv se vuelven dificiles de manejar o practicamente imposibles. En tal escenario, es mejor realizar un ataque de Payload individual.
  • Payloads individuales: Este tipo de payload es un binario que ya incluye todas las partes requeridas de meterpreter, junto con las extensiones requeridas y todas agrupadas en una sola. Cuando se invoca, la instancia de meterpreter ya tiene todo lo que necesita para funcionar y por lo tanto, metasploit no necesita perder tiempo ni ancho de banda para realizar las cargas habituales que se requieren con el enfoque por etapas.

¿Cómo utilizar meterpreter sin etapas?

En primer lugar, debemos aclarar que los 2 tipos de payloads tienen distinto formato pero esto hace que sean faciles de identificar.

Payload Por etapas Individual
Reverse TCP windows/meterpreter/reverse_tcp windows/meterpreter_reverse_tcp
Reverse HTTPS windows/meterpreter/reverse_https windows/meterpreter_reverse_https
Bind TCP windows/meterpreter/bind_tcp windows/meterpreter_bind_tcp

Ngrok.

Otra de las caracteristicas de este generador de puertas traseras es la utilización de la herramienta ngrok para generar conexiones TCP. Debemos tener autentificada una cuenta de ngrok en nuestro sistema para que funcione correctamente. Una vez teniendo indicado que tipo de Payload queremos manejar, KitHack nos solicita un LHOST y un LPORT, pero antes de estas 2 solicitudes como primera instancia KitHack nos imprime nuestra IP local, nuestra IP publica y el enlace TCP generado con ngrok. Este enlace TCP se regenera cada que nos salimos de la herramienta o cada que se ejecuta de nuevo la funcion del menú principal. Si nosotros queremos realizar un ataque fuera de nuestra red local podemos utilizar ngrok para realizar la conexión TCP con nuestra victima y asi obtener la reverse shell sin ningun tipo de problema.
Ejemplo:

Local IP: 192.168.1.223
Public IP: 201.97.221.230
Ngrok TCP: tcp://0.tcp.ngrok.io:15225

SET LHOST:
SET LPORT:

Para realizar el ataque fuera de nuestra red local, debemos colocar como LHOST: 0.tcp.ngrok.io y como LPORT el puerto que nos genere ngrok, en mi caso sería el 15225.
Una vez terminado todo el proceso, KitHack nos va a cuestionar si deseamos ejecutar msfconsole, si colocamos que si, automaticamente se va a poner Metasploit en escucha con nuestro localhost y el puerto 443 configurado. Si usted decide no ejecutar msfconsole, la sesión ngrok se perderá y por lo tanto no se va a generar la conexión. Recordemos que ngrok debe permanecer activo para que la conexión se pueda realizar y si negamos ejecutar msfconsole KitHack detiene el proceso de ngrok, por lo tanto, si vas a trabajar con ngrok es obligatorio que aceptes ejecutar msfconsole mediante KitHack.

Metodo KitHack para la certificación de APKS maliciosas.

Hemos llegado a la parte más interesante de esta documentación, este nuevo método nos permite autentificar la certificación de una APK maliciosa. ¿Alguna vez han intentado realizar un RAT con el parametro -x de msfvenom? Este método mayormente genera algún tipo de error a los usuarios ya que no admite la mayoría de APKS y tampoco realiza un método de certificación valido que pueda ser indetectable por los antivirus. El método de KitHack lo he puesto a prueba en unas 15 aplicaciones como máximo y funciona sin problemas. Hay que aclarar que no todas las aplicaciones van a ser admitidas con este proceso, tales como WhatsApp, facebook, messenger, tiktok, etc.

El proceso que realiza KitHack para estos fines es el siguiente:

  • Genera un payload de forma normal con metasploit.
  • Decompila la APK legitima proporcionada.
  • Decompila el payload que se generó anteriormente.
  • Añade nuevos permisos al archivo AndroidManifest.xml de la APK legitima.
  • Remueve el archivo MainActivity.smali de la carpeta stage del payload generado.
  • Reconfigura los archivos smali del payload generado.
  • Copia la carpeta stage del payload generado a la APK original.
  • Localiza el punto de entrada dentro del archivo AndroidManifest.xml de la APK original.
  • Modifica el archivo .smali del activity para incluir una linea que se encarga de invocar MainService para la sesión meterpreter.
  • Compila nuevamente el APK ya configurado como backdoor.
  • Certifica el archivo APK generado con jarsigner.

En mi repositorio de github se encuentra una tabla de las APKS que llegué a probar con este método y que si fueron admitidas de manera correcta, si desea echarles un vistazo da click en "View on GitHub" en la parte superior de la página.

Preguntas más frecuentes.

  1. ¿Cómo autentifico una cuenta de Ngrok?
    Primero que nada, debemos de registrarnos en la pagina oficial de ngrok con nuestro nombre, correo electronico y una contraseña. Una vez registrados y logueados en la plataforma, debemos ubicar la sección de Connect your account, ya que en ella se encontrará nuestro token para autentificarnos, copiaremos toda la instrucción del apartado negro a partir de ./ngrok authotoken y lo pegaremos en nuestra terminal en la carpeta KitHack y finalmente ejecutamos.
  2. ¿Por qué me sale el error "No module named getch"?
    KitHack esta desarrollado en la version 2.7 de python, por lo tanto, es esencial tener instalada esa versión para que se pueda ejecutar correctamente. En mi repositorio de Github se encuentran los pasos de instalación para poder instalar KitHack de manera correcta.
  3. ¿Qué tal funciona KitHack en la version 2020 de Kali linux?
    Anteriormente KitHack estaba generando un par de problemas en esta nueva versión de kali linux 2020 pero ya se ha desarrollado la solución y ya está implementada en mi repositorio de KitHack.
